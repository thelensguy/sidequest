What this is

SideQuest — a gamified job application tracker, built as a Chrome extension (Manifest V3, React + TypeScript, Vite + CRXJS). Full spec lives in PRD.md at the repo root — read that first for scope, v1 boundaries, and the "definition of done." Repo: /Users/binhthan/Documents/Projects/project/SideQuest, pushed to github.com/thelensguy/sidequest.

Core loop: capture a job posting (browser popup or an in-page floating bubble on LinkedIn/Indeed/ZipRecruiter) → track it on a dashboard → gamification (XP, levels, badges, a weighted reward dial) layered on top of the same event log, no separate backend — everything persists to chrome.storage.local.

Architecture

Shared foundation (do not casually modify — everything else depends on this contract)

  • src/lib/types.ts — JobEntry, AppEvent, ApplicationStatus, LootTableEntry. XP/levels/badges are always derived from the AppEvent log, never stored as separate counters — this is why they can't drift out of sync.
  • src/lib/storage.ts — the only file that talks to chrome.storage.local. addJobEntry/appendEvent auto-generate an id if the caller omits one (consolidated from 3 duplicate implementations during merge). getTreats/setTreats were later replaced by getLootTable/setLootTable (see Loot Table below).

Three original pieces (built in parallel, later merged)

  • Capture — src/capture/adapters/{linkedin,indeed,generic}.ts dispatched via index.ts; src/popup/ (toolbar popup, one-shot chrome.scripting.executeScript injection); src/content/ (floating in-page capture bubble + review panel, added later — runs directly in-page, no injection needed).
  • Dashboard — src/dashboard/: list/search/filter, manual add, bulk paste-import, inline edit, delete (cascades to remove that entry's events too).
  • Gamification — src/gamification/: xp.ts, levels.ts, badges.ts, wheel.ts, statusMilestones.ts (the shared anti-farming dedup primitive), GamificationPanel.tsx, RewardDial.tsx.

Build history (chronological, condensed)

1. PRD → foundation → parallel build

Interviewed for a one-page PRD, pushed back on scope (job-app tracking is current work not past, single-user not multi-tenant, one core loop not four). Built the shared foundation (manifest/build config, types, storage), then split capture/dashboard/gamification into three parallel background agents, each in its own git worktree/branch. Each piece got an independent adversarial review (a fresh agent, no attachment to the code, briefed to actively try to break it) before merge. Real bugs the reviews caught and got fixed pre-merge: a title-parsing bug in the generic capture adapter that silently dropped company names, a duplicate-save race in the popup, and a timezone bug where dates displayed a day early.

2. Merge + integration

Octopus-merged all three branches (zero real conflicts — scope discipline held). Integration work on top: consolidated id-generation into storage.ts, wired <GamificationPanel /> into the dashboard, resolved an AppEvent.type semantic mismatch between two branches (settled by checking what the sibling branch actually did, not guessing).

3. Real-world testing → bug fixes

LinkedIn adapter was broken — the original selectors guessed at semantic-looking CSS class names that don't exist; LinkedIn's actual classes are build-hashed (e.g. _69ebbc25). Fixed by having the user run a diagnostic console script on a real LinkedIn page and pasting back the actual markup — the real fix matches by href pattern (the job title is always a link to /jobs/view/<id>) instead of guessing at classes.

XP/wheel farming exploit — toggling one job's status back and forth (e.g. applied ↔ interviewing) paid out XP and crossed reward-wheel milestones on every single toggle. Fixed with a shared dedup primitive (reachedStatusKeys/countDistinctEntriesWithStatus in statusMilestones.ts) — each (job entry, status) pair only ever pays out once, no matter how many times it's re-reached. Both xp.ts and wheel.ts consume this same primitive.

Also added: delete (with event cascade), a stats bar, and a floating in-page capture bubble content-script for LinkedIn/Indeed/ZipRecruiter with an audit-before-save review panel (prevents the "captured the wrong page silently" class of bug).

4. Design exploration

Three competing dashboard directions were built as static mockups and reviewed by two adversarial design agents with different lenses (visual craft vs. UX/usability) — they disagreed on ranking, which was itself useful signal:

DirectionCraft rankUX rank
Clean Analytics (restrained SaaS style)2nd1st — real search/filter, 90% of list visible on load
RPG Quest Board (full fantasy)1st — best contrast/typography3rd — no search, no status-change UI at first, list mostly buried
Kanban (status columns)3rd2nd — column overflow at common widths

User picked RPG Quest Board on preference, then iterated: added working search/status-filter/functional status-change dropdowns (closing the UX gaps), then explicitly asked for the big spinning wheel and spacious hero panel back (had been compacted to solve the "list buried" problem) — traded that space back by compacting the quest cards and the stat-tile row instead. Several more contrast bugs caught and fixed across these iterations (the recurring pattern: a low-contrast text token reused at a newly-compacted smaller size drops below WCAG AA — check this proactively whenever font-size shrinks).

5. Pivot to "Premium HUD" direction

User: "this looks too AI/generic... I want a premium, engineering-grade interface" — explicit strict rules given: zero emoji anywhere, custom monochrome vector line icons (Lucide/Phosphor/Radix-inspired, 1.5px stroke, color-matched to text), a redesigned Reward Dial (dark #121212 face, thin 1px radial tick lines instead of colored wedges, physics-based spin), flat charcoal surfaces with sharp 1px borders (no glow/gradient), uniform quest rows with bracketed monospace status tags ([SCOUTED], [VICTORY]), and a 2px XP line instead of a thick bar. Built as a new mockup direction, verified against a computed (not eyeballed) contrast palette before writing any CSS.

6. Three new features, prototyped then ported for real

User asked for three additions — required clarifying questions before starting ("CRITICAL: ask clarifying questions first"), which resolved: EXP ticker should be both a floating popup and a HUD count-up; goal-gradient acceleration applies to both the XP line and a new wheel-unlock countdown; the loot table needed a real in-app settings screen (not just a config file) with weighted rarity tiers.

  1. "Juice" Factor — a floating +N XP ticker at the point of action, plus an animated count-up (not an instant snap) on the HUD's XP number.
  2. Goal Gradient Effect — the XP line switches to a snappier easing + pulsing glow in the last 25% of a level; the wheel gained a live countdown ("2 more applications to next spin") that pulses in accent color once down to 1 remaining.
  3. Custom Admin Loot Table — treats now have a tier (common/rare/epic) and a weight; the dial does real weighted-random selection; an editable admin panel (odds recompute live) lives on the extension's Options page.

Prototyped first as a standalone HTML mockup (plain JS, fake seed data), confirmed working, then ported into the real codebase — wired to real chrome.storage.local data, reusing the real computeXp and anti-farming dedup logic rather than reimplementing it. This is the most recent work; see "Current state" below.

Technical gotchas learned this session

  • LinkedIn/job-site DOM: don't guess at CSS class names — modern sites often ship build-hashed classes. Match by structural signal instead (e.g. an href pattern) when possible, and verify against real captured markup before writing a selector.
  • MV3 content-script injection: a function passed to chrome.scripting.executeScript({ func }) is serialized by its own source text only — it cannot close over anything outside its own body. All adapter helpers must be declared inside the function. A content script running directly in-page (like src/content/) doesn't have this restriction — it can import normally.
  • Anti-farming pattern: whenever a stat is derived from an event log keyed by (entity, state), dedupe by "first time this (entity, state) pair appeared" — see statusMilestones.ts. Don't count raw event occurrences.
  • Checkpoint races: when persisting a "last processed at N" checkpoint after an async operation (e.g. the wheel spin), re-read the live count immediately before writing — don't use a value closed over at the start of the async operation, or a concurrent event during that window gets lost/double-counted.
  • Contrast at small sizes: a muted text color that passes AA at 12px can fail at 10px on the same background. Recheck contrast whenever a compaction pass shrinks font sizes — this happened at least 4 separate times this session.
  • Artifact sandbox CSP: blocks Google Fonts / most CDN links silently (fails to a system-font fallback) — inline fonts as data URIs or just design around system font stacks for anything published via the Artifact tool.
